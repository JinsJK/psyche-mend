[SCENARIO 2 — GPU ACCELERATION OF THE VOICE PIPELINE]

Objective:
Improve performance of the multi-stage AI pipeline by enabling GPU acceleration for compute-intensive components: speech-to-text (Whisper) and text-to-speech (Coqui TTS).

Why This Scenario Was Chosen:
Scenario 1 (structured logging) revealed that STT was the dominant bottleneck at approximately 10–11 seconds per request, and TTS at approximately 2–3.5 seconds. GPU acceleration is a natural performance update to evaluate. This scenario also tests a different type of engineering effort — environment configuration and dependency management rather than code logic — which is relevant to the thesis analysis of how complexity shifts in AI-assisted development.

Scope:
- Enable GPU for Whisper (STT) and Coqui TTS
- Keep functionality unchanged
- Validate runtime GPU usage via startup prints
- Compare performance using existing structured logs (stt duration_ms, tts duration_ms, api_end duration_ms)

Baseline (Before Change):
- CPU-only execution (torch CPU build installed)
- torch.cuda.is_available() returned False
- STT approximately 10–11 seconds per request
- TTS approximately 2–3.5 seconds per request
- Whisper FP16 warning observed: "FP16 is not supported on CPU; using FP32 instead"

Changes Implemented:

1. Installed CUDA-enabled PyTorch
   Command: pip install torch==2.5.1+cu118 torchaudio==2.5.1+cu118 --index-url https://download.pytorch.org/whl/cu118
   Replaced CPU-only torch build with a CUDA-enabled build (torch>=2.5, cu118)

2. backend/speech_to_text.py
   Added explicit device selection using torch.cuda.is_available().
   Whisper model now loaded with device="cuda" or device="cpu" depending on availability.
   Startup print added: [GPU] Whisper running on: cuda

3. backend/text_to_speech.py
   GPU logic was already present (use_gpu = torch.cuda.is_available()).
   Added startup print: [GPU] TTS running on: cuda

4. main.py
   Added GPU status block at startup before backend imports:
   [GPU] CUDA available: True
   [GPU] Device: cuda

5. requirements.txt
   CPU torch lines commented out.
   GPU install command promoted to active documentation.

Measurement Method:
No changes to the logging schema. Existing structured logs (logs/app.log) are used to compare:
- stt duration_ms — before and after GPU enablement
- tts duration_ms — before and after GPU enablement
- api_end duration_ms — total request latency comparison

Work Sessions:
Session 1: ~10 minutes — dependency install, code changes to speech_to_text.py, text_to_speech.py, main.py, and requirements.txt; startup validation
Session 2: ~3 minutes — diagnosed and resolved transformers version drift caused by CUDA torch install; reinstalled transformers==4.36.2

Development Approach:
The implementation was carried out using AI-assisted development. The effort primarily involved environment configuration, dependency handling, and validation of runtime behavior. The code changes were generated by AI and verified through manual testing.

Total Time:

Iterations:

Files Changed:

Issues Faced:
1. Dependency version drift — installing torch==2.5.1+cu118 caused pip to upgrade transformers from the pinned 4.36.2 to 5.3.0 as a transitive dependency. transformers 5.3.0 added a security check (CVE-2025-32434) that blocks torch.load on versions below 2.6, preventing the sentiment model from loading and crashing the server on startup. Fixed by aligning dependency versions — either upgrading PyTorch to a compatible version (>=2.6) or reverting Transformers to a version compatible with the installed torch build. In this case, Transformers was pinned back to 4.36.2 to restore compatibility.
2. CUDA warm-up latency — the first GPU request after server start showed elevated STT latency (~4.3 s) due to CUDA context initialisation. Subsequent requests stabilised at ~1.6 s. This is expected behaviour but worth noting for thesis measurement methodology.

Outcome:
GPU acceleration was confirmed via startup prints ([GPU] CUDA available: True, [GPU] Whisper running on: cuda, [GPU] TTS running on: cuda). STT latency dropped from ~11 seconds to ~1.6 seconds and TTS from ~2.8 seconds to ~0.4–0.8 seconds. Total request latency dropped from ~16 seconds to ~3.5–4 seconds in steady state. This change altered the performance characteristics of the system without modifying its functional behaviour.

Observed Findings:

Before (CPU) — April 8 baseline:
STT     — 11,342 ms
TTS     — 2,767 ms
api_end — 15,931 ms

After (GPU) — April 15, steady-state runs (post warm-up):
STT     — ~1,520–1,638 ms  (~7x faster)
TTS     — ~374–848 ms      (~3–7x faster)
api_end — ~3,516–4,047 ms  (~4x faster)

Additional Observations:
- First GPU run (warm-up) showed STT at 4,279 ms due to CUDA context initialisation. All subsequent runs stabilised at ~1.6 seconds.
- April 15 09:02–09:06 runs showed TTS already on GPU (~1.4 s) but STT still at CPU speed (~11 s) — these were taken before the explicit device="cuda" change was applied to speech_to_text.py, confirming that TTS already had conditional GPU logic while Whisper did not.
- STT went from the dominant bottleneck (~11 s) to a minor stage (~1.6 s), changing the performance profile of the pipeline significantly.
- The system bottleneck shifted from compute-bound (STT) to network-bound (LLM), indicating that further performance improvements would require changes at the API or architectural level rather than hardware acceleration.
- Performance variability across runs highlights that GPU acceleration introduces state-dependent behaviour (e.g., warm-up effects), making benchmarking less deterministic compared to CPU execution.

Relevance to Thesis:
1. Performance Analysis — provides measurable before/after latency data for STT and TTS, the two most expensive pipeline stages.
2. Integration Complexity — shows that GPU enablement required environment-level changes (dependency install, CUDA build) rather than logic changes, which is a distinct type of integration effort.
3. Effort Redistribution — the dominant work in this scenario was environment configuration and validation, not code authoring. This supports the thesis argument that AI-assisted development shifts effort toward orchestration and system management rather than implementation.
